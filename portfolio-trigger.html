Project Details

In this project, I was responsible for ensuring data consistency and integrity across multiple related tables within a MySQL database. The goal was to automate the synchronization of records between tables upon various data operations such as insert, update, and delete. To achieve this, I implemented a series of SQL triggers tailored to the business logic and performance considerations of the transactional data flow.

TRIGGER AFTER UPDATE

Trigger Logic:

  1. The first updates general transaction metadata (created_at, updated_at, channel, cs_name, etc.) in the transactions table.
  2. The second updates product-related information (product, qty) in the product_sales table. 

CREATE TRIGGER trg_update_after_amura
AFTER UPDATE ON transaction_amura
FOR EACH ROW
BEGIN
  UPDATE transactions t
  JOIN (
    SELECT 
      transaction_id,
      created_at,
      updated_at,
      id_brand,
      channel,
      channel_type,
      user_id,
      `desc`,
      cs_name,
      price
    FROM transaction_amura
    WHERE transaction_id = NEW.transaction_id
    ORDER BY id ASC
    LIMIT 1
  ) temp ON t.transaction_id = temp.transaction_id
  SET 
    t.created_at = temp.created_at,
    t.updated_at = temp.updated_at,
    t.id_brand = temp.id_brand,
    t.channel = temp.channel,
    t.channel_type = temp.channel_type,
    t.user_id = temp.user_id,
    t.`desc` = temp.`desc`,
    t.cs_name = temp.cs_name,
    t.price = temp.price;


  UPDATE product_sales ps
  JOIN (
    SELECT 
      transaction_id,
      product,
      qty
    FROM transaction_amura
    WHERE transaction_id = NEW.transaction_id
    ORDER BY id ASC
    LIMIT 1
  ) temp_ps ON ps.transaction_id = temp_ps.transaction_id
  SET 
    ps.product = temp_ps.product,
    ps.qty = temp_ps.qty;

END

TRIGGER AFTER INSERT

Trigger Logic:

      When a new row is inserted into transaction_amura, the system checks if the transaction_id doesn't already exist in the transactions table.
      If the ID is new:
    1. The transaction data is copied into the transactions table (main transaction data).
    2. The first product from the transaction is copied into the product_sales table.
     

CREATE TRIGGER trg_after_insert_transaction_amura
AFTER INSERT ON transaction_amura
FOR EACH ROW
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM transactions WHERE transaction_id = NEW.transaction_id
  ) THEN
    INSERT INTO transactions (
      transaction_id,
      created_at,
      updated_at,
      id_brand,
      channel,
      channel_type,
      user_id,
      user_name,
      receive_name,
      handphone_number,
      address,
      `desc`,
      cs_name,
      price
    )
    VALUES (
      NEW.transaction_id,
      NEW.created_at,
      NEW.updated_at,
      NEW.id_brand,
      NEW.channel,
      NEW.channel_type,
      NEW.user_id,
      NEW.user_name,
      NEW.receive_name,
      NEW.handphone_number,
      NEW.address,
      NEW.`desc`,
      NEW.cs_name,
      NEW.price
    );

    INSERT INTO product_sales (
      created_at,
      transaction_id,
      id_brand,
      product,
      qty
    )
    VALUES (
      NEW.created_at,
      NEW.transaction_id,
      NEW.id_brand,
      NEW.product,
      NEW.qty
    );
  END IF;
END

TRIGGER AFTER DELETE

Trigger Logic:

When a row is deleted from transaction_amura, the system will:

  1. Delete the corresponding transaction from the transactions table using the same transaction_id.
  2. Delete related product data from the product_sales table using the same transaction_id.

 
CREATE trg_after_delete_transaction_amura
AFTER DELETE ON transaction_amura
FOR EACH ROW
BEGIN
  DELETE FROM transactions
  WHERE transaction_id = OLD.transaction_id;

  DELETE FROM product_sales
  WHERE transaction_id = OLD.transaction_id;
END;

Reseller Sales Data Management (Ongoing Operational Task)

This ongoing task involves managing transactional data for B2B resellers, including price changes, order revisions, new product entries, and record cleanup. I primarily work through phpMyAdmin, executing structured SQL queries to ensure that all updates are accurately reflected in the system and that data integrity is maintained across related tables. These actions are typically triggered by requests from the admin team or reseller-facing divisions, often in time sensitive scenarios where incorrect data could impact billing, delivery, or customer satisfaction.

Data Types Commonly Managed

The most frequent types of data I update or maintain include:

  • Product Pricing Adjustments — price corrections for SKUs requested by admin or pricing teams
  • Order Modifications — changes to quantity, item replacement, or cancellations due to stock or request errors
  • New Product Entries — inserting new SKUs with initial pricing and status
  • Transaction Cleanups — removing duplicate or invalid records based on verified approval
  • Status Updates — flagging orders as completed, failed, or refunded

Tools & Workflow

  • phpMyAdmin: Used to run manual SQL operations directly on live data
  • WhatsApp / Internal Ticketing: Admins typically send requests via WhatsApp or chat with specific instructions
  • SQL Queries: UPDATE, INSERT, DELETE, and some simple JOIN checks for data validation
  • Cross-checking: Double-checked against product master or CRM data to ensure consistency

Impact

While the queries are relatively simple, the impact is operationally significant:

  • Ensures accurate sales reports
  • Prevents miscommunication with resellers
  • Reduces downstream errors in logistics and billing
  • Maintains trust between teams and systems
