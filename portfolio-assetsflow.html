Project Details

This project presents the design of a streamlined data flow, database structure, and ERD for a system that enables employees to view and explore detailed information about company assets. It serves as the foundation for a reliable internal platform tailored to organizational needs.

Desc: Illustrates core entities and relationships for asset tracking, employee roles, location hierarchy, and user access. Supports end-to-end visibility, secure role-based control, and structured data flow across business units

Why I Designed the Data Flow in This Way

I built the ERD from scratch by analyzing the system requirements and aligning them with the use case design. The structure was created to ensure clear data relationships, minimize redundancy, and support scalability for future enhancements. Each entity and relationship was carefully mapped to reflect real system interactions, making the database not only efficient but also adaptable to potential upgrades as the project evolves.

Desc: Depicts interactions between Admin and Users. Admin manages data; Users access asset info and history. Includes include/extend relationships for process clarity.

Notes: This use case and ERD were developed based on discussions with the front-end team regarding current availability and technical conditions. It should be noted that both the use case and ERD may undergo future development or functional adjustments if new conditions or additional requirements arise.

Designing a Role-Based Login System for Asset Management

This project was initiated to address inefficiencies in the company’s asset management process. Previously, asset tracking and recording were done manually, which often led to errors, inconsistencies, and difficulties in locating or transferring assets. There was no proper record of asset transfers, making it harder for both IT and General Affairs (GA) to monitor equipment such as PCs, networks, desks, and chairs.

During the early stages, I faced several challenges due to a lack of coordination within my IT team. For more than a month, I did not receive clear direction or briefing from my leader or IT support, which caused delays. Additionally, my leader often emphasized only technical issues and “the simplest way out,” without considering user convenience. To ensure the system truly helps users, I decided to take initiative and consulted directly with the GA senior, who manages office assets daily. From these discussions, I identified the need for an Asset Transfer feature, which was not initially considered. I then integrated this feature into the system design so users could track asset movements more effectively.

This proactive approach helped shape a system that is not only technically functional but also user-friendly and aligned with real operational needs.

My Role in This Project

In this project, I was responsible for designing and developing the database structure to ensure data consistency, efficiency, and scalability. My main tasks included:

  • Analyzing Requirements – Interpreted business and system needs to translate them into logical database structures.
  • Designing ERD (Entity Relationship Diagram) – Built the ERD as a blueprint for the database, defining entities, relationships, and constraints.
  • Creating Use Case Scenarios – Developed use case diagrams to illustrate system interactions and data flows (requested by front end)
  • Database Implementation. – Converted ERD into a physical database using MySQL, ensuring normalization and optimized queries
  • Collaboration & Documentation – Worked closely with the team to align database design with application requirements and documented all design decisions.

This role strengthened my skills in database administration, system analysis, and problem-solving, while also improving my ability to communicate technical concepts to both technical and non-technical stakeholders.
