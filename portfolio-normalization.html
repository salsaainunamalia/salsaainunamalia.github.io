Project Details

To support the CRM team's needs in querying transactional data via SQL and Metabase, the transaction_amura table was normalized into two separate tables: transactions and product_sales. This restructuring was aimed at simplifying the data model, enabling easier joins, and improving query performance for the data analyst workflow.

Desc: This table initially served as the main transactional dataset used by the CRM team. It contains mixed data including transaction details, product info, and customer-related fields—all in a single wide table.

Purpose:It was the raw table used for storing all transaction-related data but lacked structure for scalable querying and maintenance.

Desc: This table was created to isolate the core transactional information such as transaction_id, timestamps, brand ID, channel, customer identifiers, and price.

Purpose: Acts as the main table representing each unique transaction. Makes it easier for data analysts to focus on transaction-level queries without product-level noise.

Desc: This table stores product-level information related to each transaction, such as product ID and quantity, with a reference to the transaction_id.

Purpose: Separates product sales data for each transaction, allowing one-to-many relationships between a transaction and the products involved—essential for detailed sales reporting and aggregation.

Project Background

In the CRM division of my company, the data analyst had just transitioned into a new role, and was starting to work with SQL for the first time—previously relying on Python for data wrangling. Meanwhile, the CRM system was led by a newly assigned manager, and things were still getting structured.

The data analyst needed a clean and reliable schema to support query efficiency, especially for operations like table joins and filtering in Metabase dashboards. That's when I stepped in.

My Role

As a Database Administrator, I was responsible for improving data structure through normalization. I worked directly with the CRM team to review their raw transactional data, which was initially stored in a single master table: transaction_amura. After evaluating the schema, I designed and implemented a more normalized version of the database—breaking it into two separate but relationally structured tables: transactions and product_sales.

What i did?

  • Analyzed the original schema to identify repeating groups and redundant columns
  • Normalized the data up to the Second Normal Form (2NF) to eliminate data duplication
  • Created primary keys and foreign keys for proper relational mapping
  • Improved readability and usability for analysts by optimizing table relationships
  • Maintained data integrity while restructuring
  • Supported analyst workflows by ensuring easier joins and cleaner querying

Outcome

This restructuring helped the CRM data analyst access necessary fields for analysis faster and more intuitively. With a well-structured schema, their SQL learning process became smoother, and the dashboards in Metabase loaded more efficiently. This also laid the groundwork for more scalable reporting in the CRM team moving forward.

Key Skills Applied

  • Database Normalization (1NF → 2NF)
  • Schema Design
  • SQL (DDL/DML)
  • Stakeholder Communication
  • Performance-aware Structuring
  • Cross-team collaboration with CRM & Data Analyst

Takeaway

"Good data isn’t just about having a lot of it. It’s about making sure it’s structured, accessible, and scalable."

This project highlighted how behind-the-scenes structuring work can massively impact how smooth things run on the front end, especially when supporting team members new to SQL.
